# DXN Project Plan

This document tracks the **project plan**, **completed vs outstanding work**, and the **steps required to build and integrate dxn-setup** (the project bootloader that initializes a vault server/app) with this repository.

**Status legend:** ✅ Completed · 🟡 In progress · 🔴 Not started

---

## 1. Overview

- **dxn-setup** is the **one-time provisioning CLI** (bootloader) that initializes a DXN vault server/app. It derives a vault key, preloads secrets, generates `config.json`, and writes a provisioning lock so the instance cannot be re-initialized.
- **dxn-core** is the runtime server. It must start only after provisioning: it checks for `config.json` and `.dxn-setup-lock.json`, verifies the vault key, and performs one-time schema/seed from config, then treats DB and vault as source of truth.

Specs and flows are in `z.documents/`:

- `dxn-setup-spec.md` – CLI contract, files written, exit codes.
- `dxn-setup-flow.md` – step-by-step provisioning flow.
- `dxn-core-init-flow.md` – server startup and first-run initialization.

## 2. Build and Integrate dxn-setup

### 2.1 What dxn-setup Is

- **Rust crate** producing a single executable: `dxn-setup`.
- **Single-use per instance**: if `.dxn-setup-lock.json` exists in `dxn-core/`, it exits with code `10` and does not re-provision.
- **Responsibilities**:
  1. Collect username and shared secret (interactive or `DXN_SETUP_USERNAME` / `DXN_SETUP_SECRET` when `--non-interactive`).
  2. Derive vault master key (Argon2id/Balloon, configurable KDF profile).
  3. Preload vault key/value pairs from `--secrets-file`.
  4. Generate `dxn-core/config.json` from `--config-template` + optional `--settings-file`.
  5. Write `dxn-core/.dxn-setup-lock.json` (instance_id, project_root, created_at, created_by, config_checksum, vault metadata including KDF params, salt, key_fingerprint).

### 2.2 Steps to Build dxn-setup

1. **Create the crate**
   - From repo root: `cargo new --bin dxn-setup` (or add `dxn-setup/` with `Cargo.toml` and `src/main.rs`).
   - In workspace `Cargo.toml` (if present), add `dxn-setup` to `[workspace].members`.
   - Set `edition = "2021"`, `rust-version`, and metadata (name, description, license) in `dxn-setup/Cargo.toml`.

2. **Implement CLI**
   - Parse args: `--project-root`, `--config-template`, `--settings-file`, `--secrets-file`, `--kdf-profile`, `--non-interactive`.
   - Resolve paths (absolute), validate:
     - `project-root` exists.
     - `<project-root>/dxn-core` exists.
     - Template, settings, and secrets files exist and are readable.
   - If `dxn-core/.dxn-setup-lock.json` exists → print message, exit code `10`.

3. **Collect identity and secret**
   - If not `--non-interactive`: prompt for Username and Shared secret (no echo).
   - If `--non-interactive`: read `DXN_SETUP_USERNAME` and `DXN_SETUP_SECRET` from env.
   - Never log or store the plaintext secret.

4. **Derive vault key**
   - Implement KDF (e.g. Argon2id) with profile (memory, iterations, parallelism).
   - Generate random salt; derive key; compute `key_fingerprint = HMAC-SHA256(derived_key, "DXN_VAULT")`.
   - Dependencies: e.g. `argon2`, `rand`, `hmac`, `sha2`.

5. **Vault storage**
   - Create/open `dxn-core/vault.db` (or path under project-root).
   - Encrypt and store each key/value from `--secrets-file` (format e.g. `path.to.key=value`).
   - Use derived key for encryption; do not persist raw key or secret.

6. **Generate config.json**
   - Load `--config-template` (JSON).
   - If `--settings-file` provided, load and merge (e.g. host, ports, base URL).
   - Optionally validate (required fields, no conflicting routes/models).
   - Write result to `dxn-core/config.json`.

7. **Write provisioning lock**
   - Compute `config_checksum = sha256(config.json)`.
   - Write `dxn-core/.dxn-setup-lock.json` with: instance_id (UUID), project_root, created_at, created_by, config_checksum, vault metadata (path, KDF params, salt, key_fingerprint).
   - Use exit codes per spec: 0 success; 10 already provisioned; 11–15 for specific errors.

8. **Tests and docs**
   - Unit tests for KDF, fingerprint, and config merge.
   - Integration test: run dxn-setup in a temp dir, assert `config.json` and `.dxn-setup-lock.json` exist and lock prevents second run (exit 10).
   - README in `dxn-setup/` with usage and exit codes.

### 2.3 Steps to Integrate dxn-setup With This Project

1. **dxn-core startup checks**
   - On startup, before loading config or opening DBs:
     - If `config.json` is missing → exit with clear error: *"config.json not found. This file must be generated by dxn-setup."*
     - If `.dxn-setup-lock.json` is missing → exit with clear error: *"DXN core is not provisioned. Run dxn-setup before starting the server."*
   - Implement this in `dxn-core/src/main.rs` (or a small `bootstrap` module) at the very beginning of `main`.

2. **Vault key verification (dxn-core)**
   - Read vault path, KDF params, salt, and `key_fingerprint` from `.dxn-setup-lock.json`.
   - Obtain shared secret at runtime (e.g. `DXN_CORE_SECRET` env var).
   - Derive key using same KDF; compute fingerprint; compare with stored value.
   - If mismatch → exit: *"Invalid shared secret for this server instance."*
   - Open vault (e.g. `vault.db`) with derived key for runtime secret access.

3. **One-time initialization state**
   - Add `.dxn-core-state.json` concept:
     - If missing or `initialized != true`: run first-time init (create schema from config, register functions/routes, apply initial migrations), then write `.dxn-core-state.json` with initialized, instance_id, initialized_at, config_checksum, db_migration_version.
     - If present: compare current `config.json` checksum with stored one; optionally warn or fail if config changed (per `dxn-core-init-flow.md`).
   - Ensure dxn-core does not re-seed or re-create schema from config on every boot after first run.

4. **Repo layout and docs**
   - Ensure project root contains (or references) a `config.template.json` and example `settings.json` / `secrets.txt` for running dxn-setup.
   - Update root `README.md` with:
     - "First-time setup: run dxn-setup from the project root (see dxn-setup/README.md)."
     - "Then start the server: cd dxn-core && cargo run (with DXN_CORE_SECRET set)."
   - In `z.documents/`, keep `dxn-setup-spec.md`, `dxn-setup-flow.md`, and `dxn-core-init-flow.md` as the authoritative flows; link to them from README and plan.md.

5. **CI (optional but recommended)**
   - Build `dxn-setup` in CI: `cargo build -p dxn-setup`.
   - Optional: run integration test that provisions a temp directory with dxn-setup and then starts dxn-core (with mock secret) to verify init flow.
