# First-run initialization (dxn-core after `dxn-setup`)

*End-to-end contract with `dxn-setup-todo.md` Phase D (staging) and Phase D2 (core). Setup writes `dxn-core/.dxn-keystore-seed/` (or configured path); **dxn-core** must ingest into the SecureStore vault then remove staging.*

## What “initialize” means on first boot

1. **Resolve paths** — `settings.keystoreSeedPath` in `config.json` and/or `keystore_seed_path` in `.dxn-setup-lock.json` (define precedence: e.g. lock wins, or config-only after merge).
2. **Keystore vault** — Create or open the encrypted keystore at the configured vault path (see `examples/z.documents/keystore.md` and `system::keystore`).
3. **Ingest staged seeds** — If the seed directory exists and contains `secrets.seed` + `manifest.json` (per setup format): validate manifest vs file, `set` each key into `SecretsManager`, persist (`save` / `save_as`).
4. **Remove staging** — After successful persist, **delete the entire seed directory** (or move to quarantine then delete). Staging is plaintext-at-rest until then.
5. **Record completion** — Write or update `.dxn-core-state.json` (or equivalent flag) so later boots do not re-ingest or require the seed dir.
6. **Idempotence** — Second boot: if state says ingested **or** seed dir is absent, **no-op** (do not fail). If first run failed mid-ingest, allow **retry** while seed dir still exists.

## Phase G2: Keystore seed ingest (implementation tasks)

- [ ] Read staging path from config + lock; resolve relative paths against `config.json` parent or `project_root`
- [ ] If seed directory missing: skip (already done or never staged)
- [ ] If seed directory present: parse `manifest.json` + `secrets.seed` (same rules as setup: skip blanks/`#`, key shape)
- [ ] Open/create SecureStore vault + key per config; ingest all entries; persist vault
- [ ] **Only after success:** recursively remove `.dxn-keystore-seed` (or configured path)
- [ ] On ingest failure: **do not** delete staging; log clearly; exit non-zero or refuse to serve (product choice)
- [ ] Mark ingest complete in `.dxn-core-state.json` (or nested field) tied to instance / checksum
- [ ] Tests: temp dirs — staged seeds → ingest → staging gone → second run no-op

## Startup behavior: run every time vs only once

| Approach | Behavior | Pros | Cons |
|----------|------------|------|------|
| **A — Full pipeline every boot** | Always run schema bootstrap, seed ingest checks, etc. | Simple mental model | Wastes work; risk of repeating side effects unless every step is strictly idempotent |
| **B — Heavy init only once** | Use `.dxn-core-state.json` (or “initialized” flag). First boot runs ingest + bootstrap; later boots skip | Fast, predictable, matches “provision once” | Must maintain state file; need migration if init steps change |
| **C — Recommended hybrid** | **Every startup:** cheap checks (paths, state file, “does seed dir exist?”). **Expensive work** (ingest, full re-seed from config) **only when** state says not initialized **or** seed dir exists (retry after failed first run) **or** explicit admin “reinit” flag (future) | Safe retries; normal boots stay fast; no duplicate ingest after success | Slightly more branching in `main` / bootstrap module |

**Recommendation:** **C** — run a **lightweight init routine on every startup** that exits quickly when `initialized == true` and no seed directory is present; run **full first-run** only when needed. Never re-delete or re-ingest blindly without checking state + filesystem.
